# Decision journal — 24


## Decision
Decided on flat fields (maxTextLength, outputFormats, supportsStreaming) directly on the Provider object rather than a nested `capabilities` sub-object. Rationale: the steering capability is already handled per-voice via the existing `steering` descriptor, so a nested `capabilities` schema would be mostly empty for steering — flat fields on Provider + per-voice `steering` together cover the full ask more cleanly. JobStatus narrowed to the three values (`completed`, `pending`, `failed`) the API actually returns today rather than the five in the original plan.
